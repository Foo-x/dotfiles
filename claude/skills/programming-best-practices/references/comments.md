# コメント

## 基本原則：Why を書き、What は書かない

コメントはコードが「何をしているか」ではなく「なぜそうしているか」を説明する。「何をしているか」は適切な命名と構造を持つコードそのものが語るべきであり、コードと同じ内容をコメントで繰り返すのは冗長で、修正漏れによる乖離の原因にもなる。

## コメントに含めるべき内容

- **実装意図**: なぜこの実装・設計を選んだか（他の選択肢ではなくこれを選んだ理由、トレードオフ）
- **非自明な制約や前提**: 一見不要に見えるが実は必要な処理、外部要因（バグ回避・パフォーマンス要件・仕様上の制約）に起因する実装
- **複雑なロジックの説明**: アルゴリズムや計算式など、コードを読むだけでは意図を追いにくい処理の要約
- **クラス・関数の責務**: 公開インターフェースにおいて、何をするものか（How ではなく What/責務のレベル）
- **危険な使い方への警告**: 副作用、スレッドセーフでない、呼び出し順序に制約がある、などコンパイラ／型システムで表現できない注意点

## 書かないコメント

- コードを読めばすぐにわかること（変数への代入をそのまま説明する、など）
- コメントアウトされた古いコード（バージョン管理履歴に任せて削除する）
- 現在のタスク・issue・呼び出し元への言及（「〇〇機能のために追加」「issue #123 の対応」など）。コードが変わるにつれ陳腐化するため、そうした文脈は PR やコミットメッセージに書く

## コードを改善する方が優先

わかりにくいコードにわかりやすいコメントを付けるのではなく、命名・関数分割・構造の見直しによってコード自体をわかりやすくすることを優先する。コメントは「コードでは表現しきれない情報」を補うために使う。

要件・仕様自体が本質的に複雑で、コードを整理してもなお理解しにくさが残る場合は、その複雑さを補うためにわかりやすいコメントを付ける。

## コメントの保守

コメントはコードと一緒に更新する。コードとコメントの内容が食い違っている状態は、コメントがない状態より悪い（読み手を誤誘導する）。変更に伴い意味を失ったコメントは更新するか削除する。
