# プログラミング原則

## SOLID原則

### 1. Single Responsibility Principle（単一責任原則）
クラスは変更の理由を1つだけ持つべきである。

### 2. Open/Closed Principle（オープン・クローズドの原則）
ソフトウェアエンティティは拡張に対して開いており、修正に対して閉じているべき。

### 3. Liskov Substitution Principle（リスコフの置換原則）
派生クラスは基底クラスと置換可能でなければならない。

### 4. Interface Segregation Principle（インターフェース分離の原則）
クライアントは使用しないメソッドに依存すべきではない。

### 5. Dependency Inversion Principle（依存性逆転の原則）
上位モジュールは下位モジュールに依存すべきではない。両者は抽象に依存すべき。

## 基本原則

### DRY（Don't Repeat Yourself）
知識の重複を避け、単一の信頼できる情報源を持つべき。

### KISS（Keep It Simple, Stupid）
設計はできるだけシンプルに保つべき。

### YAGNI（You Aren't Gonna Need It）
現在必要でない機能は実装すべきではない。

## 設計原則

### 関心の分離（Separation of Concerns）
プログラムを異なる関心事に分割し、各部分が1つの関心事のみを扱う。

### デメテルの法則（Law of Demeter）
オブジェクトは直接の知り合いとのみ会話すべき。

### Composition over Inheritance（継承よりコンポジション）
クラスの再利用は継承よりもコンポジションを優先すべき。

### 純粋関数（Pure Functions）
関数は副作用を持たず、同じ入力に対して常に同じ出力を返すべき。

## パッケージ設計の原則

### パッケージ凝集度の原則

| 原則 | 定義 |
|------|------|
| **REP（再利用・リリース等価）** | 再利用の単位はリリースの単位と一致すべき |
| **CRP（共通再利用）** | 一緒に再利用されるクラスを同じパッケージに |
| **CCP（共通閉鎖）** | 同じ理由・タイミングで変更されるクラスを同じパッケージに |

### パッケージ結合度の原則

| 原則 | 定義 |
|------|------|
| **ADP（非循環依存関係）** | パッケージの依存関係グラフに循環があってはならない |
| **SDP（安定依存）** | 不安定なパッケージから安定したパッケージへ依存すべき |
| **SAP（安定抽象化）** | パッケージの抽象度は安定度に比例すべき |
