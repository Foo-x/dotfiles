# レビューチェックリスト（8観点）

このファイルは `requirements-reviewer` スキルが参照する詳細チェックリストです。

---

## 観点 1: 曖昧性の検出

曖昧な表現は、実装者ごとに異なる解釈を生み、仕様書が書けない原因となります。

### チェック項目

- [ ] 測定不能な修飾語を含む要求はないか（「適切に」「迅速に」「十分に」「なるべく」）
- [ ] 主語が省略されており、アクターが特定できない要求はないか
- [ ] 「など」「その他」で重要な対象を省略していないか
- [ ] 条件の範囲が曖昧（「一定期間」「多数の」「大きい場合」）ではないか
- [ ] 同一概念に複数の名称が使われていないか（`references/ambiguity-dictionary.md` を参照）

### 良い例 / 悪い例

| 悪い例 | 問題 | 良い例 |
|--------|------|--------|
| `システムは適切にエラーを処理しなければならない` | 「適切に」が測定不能 | `WHEN エラーが発生した時、システムはエラーコードと原因メッセージを画面に表示しなければならない` |
| `処理はできるだけ早く完了しなければならない` | 「できるだけ早く」に数値基準がない | `処理は 3 秒以内に完了しなければならない` |
| `大きすぎるファイルはアップロードできない` | 「大きすぎる」の定義がない | `10 MB を超えるファイルはアップロードできない` |
| `ユーザーは設定を変更できる` | アクターの権限範囲が不明 | `認証済みユーザーは自身のプロフィール設定を変更できる` |

### 典型的な問題パターン

- **パターン A**: 副詞による曖昧性 — 「素早く」「安全に」「正しく」「効率的に」
- **パターン B**: 境界値の未定義 — 「多い場合」「長時間」「大量の」
- **パターン C**: 主語・アクターの省略 — 「表示される」（誰が？）「更新できる」（誰が？）
- **パターン D**: 接続詞による複数解釈 — 「AまたはB」（排他的OR か包含的OR か）

## 観点 2: 一貫性

要求間の矛盾や用語の揺れは、仕様書の品質を根本から損ないます。

### チェック項目

- [ ] 同一の概念が異なる用語で記述されていないか（ユーザー/利用者/操作者 等）
- [ ] 2つ以上の要求が互いに矛盾していないか
- [ ] ある要求が別の要求の前提を満たせないケースはないか
- [ ] EARS記法を使用している場合、同一イベントに対して矛盾する応答が定義されていないか
- [ ] 番号・ID体系が一貫しているか

### 良い例 / 悪い例

| 悪い例 | 問題 | 改善方針 |
|--------|------|---------|
| REQ-01「ユーザーはパスワードを変更できる」 / REQ-07「パスワードは変更不可とする」 | 直接矛盾 | どちらかを削除、または条件を明記 |
| 「利用者」「ユーザー」「操作者」が混在 | 用語の揺れ | 用語定義セクションで統一、以降一貫して使用 |
| 「セッションは30分で終了」と「ユーザーがアクティブな間はセッションを維持」が共存 | 暗黙的矛盾 | アクティビティの定義と組み合わせ方を明記 |

### 典型的な問題パターン

- **パターン A**: 直接矛盾 — 同一主語・同一条件に対して逆の動作が要求される
- **パターン B**: 前提の循環 — REQ-A は REQ-B が満たされた場合に限り成立するが、REQ-B は REQ-A を前提とする
- **パターン C**: 用語の揺れ — 同一エンティティが文脈によって異なる名称で呼ばれる
- **パターン D**: 暗黙的矛盾 — 直接は矛盾しないが、特定のシナリオで両立不可能

## 観点 3: 完全性

必要な要求が抜けている状態では、仕様書に穴が生じます。

### チェック項目

- [ ] 主要なユースケースの正常系がすべて記述されているか
- [ ] 主要な異常系・エラーケースが記述されているか
- [ ] すべての要求に前提条件（Pre-condition）が明示されているか（暗黙でなく）
- [ ] EARS 記法の場合: WHEN に対する IF...THEN（エラー時の振る舞い）が定義されているか
- [ ] 複数のアクターが存在する場合、各アクターの要求が網羅されているか
- [ ] 外部システム・APIとの連携要求が記述されているか
- [ ] データの永続化・削除・アーカイブに関する要求が含まれているか

### 良い例 / 悪い例

| 悪い例（抜け漏れ） | 良い補足 |
|-------------------|---------|
| ログイン成功時の動作のみ記述、失敗時の動作がない | `IF 認証が失敗した場合、システムはエラーメッセージを表示し、3回失敗でアカウントをロックしなければならない` |
| ファイルアップロードの正常系のみ、ネットワーク断時の動作がない | `IF アップロード中にネットワーク接続が失われた場合、システムはユーザーに通知し、再試行オプションを提供しなければならない` |
| 管理者向け要求のみで、一般ユーザーの要求が未定義 | 各アクターの要求セクションを分けて定義 |

### 典型的な問題パターン

- **パターン A**: ハッピーパスのみ — 正常系は詳細だが、異常系・エッジケースが皆無
- **パターン B**: アクターの欠落 — 1アクターの要求は詳細だが、他のアクターが未定義
- **パターン C**: 境界イベントの欠落 — 開始・終了・タイムアウトが未定義
- **パターン D**: 依存関係の欠落 — 外部システム連携が要求に含まれていない

## 観点 4: テスト可能性

検証できない要求は、受け入れ基準として機能しません。

### チェック項目

- [ ] 各要求に対して「どのようにテストするか」を説明できるか
- [ ] 合否を客観的に判定できる基準があるか（数値・状態・メッセージ等）
- [ ] 「成功した場合」「正しく処理された場合」等の主観的成功基準が使われていないか
- [ ] 非機能要求（パフォーマンス等）に数値基準が含まれているか
- [ ] 「すべての」「常に」等の全称量化子は、実際にテスト可能な範囲か

### 良い例 / 悪い例

| 悪い例 | テスト困難な理由 | 良い例 |
|--------|----------------|--------|
| `システムは正しく動作しなければならない` | 「正しく」の基準がない | 各機能を個別の検証可能な要求に分解 |
| `パフォーマンスを改善しなければならない` | ベースラインと目標値がない | `95パーセンタイルのレスポンスタイムが 500ms 以内でなければならない` |
| `ユーザーエクスペリエンスを向上させる` | 主観的で測定不能 | `ユーザーがタスクを完了するまでのクリック数が 5 回以下でなければならない` |
| `セキュリティを確保しなければならない` | 「確保」の定義がない | `パスワードは bcrypt で強度 12 以上でハッシュ化して保存しなければならない` |

### 典型的な問題パターン

- **パターン A**: 成果の混同 — 「処理が完了する」は確認できるが、「うまく処理される」は確認できない
- **パターン B**: 相対的な基準 — 「現行より速く」（現行の計測値がなければテスト不可）
- **パターン C**: 無限大の全称量化 — 「すべてのユーザー入力を処理する」（入力の全量が不明）
- **パターン D**: 過程の要求 — 「ログを記録しながら処理する」（ログの存在はテスト可能だが、処理の正しさは別）

## 観点 5: トレーサビリティ

各要求が、解決すべき問題・達成すべき価値に結びついているかを確認します。

### チェック項目

- [ ] ドキュメント内に目的・背景・解決すべき問題が記述されているか
- [ ] 各要求が何の目的のために存在するかが理解できるか
- [ ] 孤立した要求（目的との接続が不明な要求）はないか
- [ ] ビジネス要求（なぜ必要か）とシステム要求（何をするか）が区別されているか
- [ ] 要求 ID が定義されており、後続ドキュメントで参照可能か

### 良い例 / 悪い例

| 悪い例 | 問題 | 改善方針 |
|--------|------|---------|
| 目的・背景セクションがなく、要求のリストのみ | 各要求の存在理由が不明 | 概要セクションに目的・スコープ・ステークホルダーを記述 |
| `システムはダークモードをサポートしなければならない`（背景なし） | なぜダークモードが必要かが不明 | `[目的: アクセシビリティ向上、視覚的疲労の軽減]` を付記 |
| ID なしの要求リスト | spec・テストケースからの参照不可 | `REQ-001` 等の ID を付与 |

### 典型的な問題パターン

- **パターン A**: 目的なき要求 — 「なぜ必要か」がドキュメント内で言及されていない
- **パターン B**: ID の欠如 — 後続ドキュメントで追跡できない
- **パターン C**: ビジネス・システム要求の混在 — 「売上向上のため○○する」と「○○ボタンをクリックすると」が同列

## 観点 6: 粒度

1つの要求には1つの責務のみを含めます。

### チェック項目

- [ ] 1つの要求文に複数の独立した機能が含まれていないか（「AかつB」「AまたはB」）
- [ ] 「and」「また」「さらに」で複数の要求が連結されていないか
- [ ] 逆に、過度に細分化されて文脈が失われていないか
- [ ] 要求の粒度が全体を通して一貫しているか（一部が非常に詳細で他が大まか等）

### 良い例 / 悪い例

| 悪い例 | 問題 | 改善方針 |
|--------|------|---------|
| `ユーザーはプロフィールを更新し、画像をアップロードし、SNS連携を設定できる` | 3つの独立した機能が混在 | 3つの要求に分割: REQ-A（プロフィール更新）, REQ-B（画像アップロード）, REQ-C（SNS連携） |
| `WHEN ボタンが押された時、ログを記録し、データを保存し、メールを送信し、画面を更新する` | 4つの副作用が1要求に混在 | 各副作用を独立した要求（またはサブ要求）に分割 |
| 全体の80%は高粒度な詳細要求だが、「認証機能を実装する」が1行のみ | 粒度の不一致 | 重要な機能（認証）を同等の粒度で記述 |

### 典型的な問題パターン

- **パターン A**: AND 汚染 — 1文に「かつ」で結合された複数の独立した要求
- **パターン B**: Epic 要求 — 「○○システムを実装する」という粗すぎる要求
- **パターン C**: 粒度の揺れ — 文書全体で粒度が一貫していない

## 観点 7: 過剰仕様回避

実装詳細・不要な制約は、設計の自由度を損ない、柔軟性を失わせます。

### チェック項目

- [ ] 実装言語・フレームワーク・DBの種類等、技術選定を要求が縛っていないか
- [ ] 「配列で」「ループを使って」等のアルゴリズム実装詳細が含まれていないか
- [ ] UIのレイアウト・配色等の詳細デザインが機能要求に混入していないか（別ドキュメントへ）
- [ ] 「現行と同じ方法で」等、既存実装への依存が含まれていないか
- [ ] 技術的制約として記述されているが、実際にはビジネス制約である要求はないか

### 良い例 / 悪い例

| 悪い例 | 問題 | 良い例 |
|--------|------|--------|
| `PostgreSQL を使用してユーザーデータを保存しなければならない` | DB技術を要求が縛っている | `ユーザーデータは永続化されなければならない`（技術選定は設計フェーズで） |
| `画面左上の青いボタンを押すと保存される` | UIレイアウトが要求に混入 | `ユーザーが保存操作を実行した時、データが保存されなければならない` |
| `React hooks を使って状態を管理する` | フロントエンド実装詳細 | `ユーザーの操作に応じて画面が即座に更新されなければならない` |

### 典型的な問題パターン

- **パターン A**: 技術バイアス — 特定の技術スタックへの言及
- **パターン B**: UI ウォーターフォール — 要求に詳細な画面設計が混入
- **パターン C**: アルゴリズム指定 — 「バブルソートで」「再帰で」等の実装手法の指定

## 観点 8: 非機能要求の分離

機能要求と非機能要求を同一セクションに混在させると、仕様書の管理が困難になります。

### チェック項目

- [ ] パフォーマンス要求（レスポンスタイム・スループット）が機能要求セクションに混在していないか
- [ ] セキュリティ要求（暗号化・認証強度）が機能要求の条件として埋め込まれていないか
- [ ] 可用性・信頼性要求（稼働率・RTO・RPO）が独立したセクションにあるか
- [ ] ログ・監査要求が機能要求の注釈として埋め込まれていないか
- [ ] スケーラビリティ要求が記述されており、適切なセクションにあるか

### 非機能要求の分類（参考）

| カテゴリ | 例 |
|---------|-----|
| **パフォーマンス** | レスポンスタイム・スループット・同時接続数 |
| **セキュリティ** | 認証方式・暗号化強度・アクセス制御 |
| **可用性** | 稼働率（例: 99.9%）・計画停止時間 |
| **信頼性** | RTO（目標復旧時間）・RPO（目標復旧時点） |
| **保守性** | ログ・監査証跡・デバッグ容易性 |
| **移植性** | 対応ブラウザ・OS・デバイス |
| **スケーラビリティ** | ユーザー数・データ量の成長への対応 |

### 典型的な問題パターン

- **パターン A**: 性能条件の埋め込み — `WHEN ボタンが押された時、**500ms 以内に**処理を完了する` （非機能を機能要求の条件に混入）
- **パターン B**: セキュリティ混入 — `ユーザーがログインする（パスワードは bcrypt でハッシュ化）`
- **パターン C**: 非機能セクションの欠如 — ドキュメントに非機能要求セクションが一切ない
