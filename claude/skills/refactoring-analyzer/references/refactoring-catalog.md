# リファクタリング手法カタログ

コードスメルを解消するための具体的なリファクタリング手法を体系的に分類したカタログです。

---

## Composing Methods（メソッドの構成）

### Extract Method（メソッドの抽出）
**目的**: 意味のあるまとまりをメソッドとして切り出し、元のメソッドを短く読みやすくする。
**手順の概要**: まとまった処理を新しいメソッドに移動し、処理内容を表す名前を付け、元の場所から呼び出す。
```
// Before
function printOrder(order) {
  console.log(`Order: ${order.id}`);
  let total = 0;
  for (const item of order.items) total += item.price * item.qty;
  console.log(`Total: ${total}`);
}

// After
function printOrder(order) {
  console.log(`Order: ${order.id}`);
  console.log(`Total: ${calculateTotal(order)}`);
}
function calculateTotal(order) {
  return order.items.reduce((sum, item) => sum + item.price * item.qty, 0);
}
```

### Inline Method（メソッドのインライン化）
**目的**: メソッド本体が呼び出し元より分かりやすい場合に、間接層を取り除く。
**手順の概要**: メソッド呼び出しをメソッド本体で置き換え、メソッド定義を削除する。

### Replace Temp with Query（一時変数のクエリ化）
**目的**: 一時変数への代入をメソッド呼び出しに置き換え、重複計算を排除し抽出をしやすくする。
**手順の概要**: 一時変数に代入している式をメソッドとして抽出し、変数の参照をメソッド呼び出しに置き換える。

### Replace Method with Method Object（メソッドオブジェクトへの置換）
**目的**: ローカル変数が多く分割しづらい長いメソッドを、専用オブジェクトに移して自由に分解できるようにする。
**手順の概要**: メソッドとそのローカル変数をフィールドに持つ新クラスを作り、メソッド本体をそのクラスの中でさらに Extract Method する。

### Substitute Algorithm（アルゴリズムの置き換え）
**目的**: より明快なアルゴリズムに置き換える。
**手順の概要**: 新しいアルゴリズムを実装したメソッドを用意し、テストで同値性を確認してから差し替える。

## Moving Features Between Objects（オブジェクト間での機能の移動）

### Move Method（メソッドの移動）
**目的**: メソッドが自身のクラスより他クラスのデータを多く使っている場合に、適切なクラスへ移す。
**手順の概要**: 移動先クラスに同等のメソッドを作成し、元のメソッドを委譲または削除する。

### Move Field（フィールドの移動）
**目的**: フィールドが自身のクラスより他クラスで多く使われている場合に、適切なクラスへ移す。
**手順の概要**: 移動先クラスにフィールドを作り、参照元をすべて新しいフィールドに向け直す。

### Extract Class（クラスの抽出）
**目的**: 1つのクラスが担う複数の責任を、それぞれ独立したクラスに分離する。
**手順の概要**: 分離すべき責任に対応するフィールド・メソッドをまとめて新クラスに移し、元クラスから委譲する。

### Inline Class（クラスのインライン化）
**目的**: ほとんど何もしていないクラスを他クラスに統合し、無駄な間接層を無くす。
**手順の概要**: 対象クラスの全メンバーを別クラスに移し、元クラスを削除する。

### Hide Delegate（委譲の隠蔽）
**目的**: クライアントが内部オブジェクトの構造を知りすぎないよう、委譲メソッドで隠蔽する。
**手順の概要**: 委譲先が持つメソッドと同名のラッパーメソッドをサーバー側に作り、クライアントはサーバーのみを呼び出す。

### Remove Middle Man（仲介者の除去）
**目的**: 単純な委譲しかしていないクラスを取り除き、クライアントが実オブジェクトを直接呼ぶようにする。
**手順の概要**: クライアントに実オブジェクトを取得するメソッドを提供し、不要になった委譲メソッドを削除する。

## Organizing Data（データの整理）

### Encapsulate Field（フィールドのカプセル化）
**目的**: publicフィールドを直接公開せず、getter/setter経由でアクセスさせる。
**手順の概要**: フィールドをprivateにし、アクセス用のgetter/setterを追加、参照箇所を置き換える。

### Replace Data Value with Object（データ値のオブジェクト化）
**目的**: 単なるプリミティブ値として扱われているドメイン概念を、専用オブジェクトとして表現する。
**手順の概要**: 値を保持するクラスを作り、フィールドの型をそのクラスに変更、関連ロジックをクラス内に移す。

### Introduce Value Object（値オブジェクトの導入）
**目的**: 等価性で比較される不変な値をオブジェクトとして表現し、関連バリデーションをまとめる。
**手順の概要**: イミュータブルな値オブジェクトを作成し、等価比較・バリデーションをその中に実装する。

### Change Bidirectional Association to Unidirectional（双方向関連の単方向化）
**目的**: 不要な双方向依存を解消し、結合度を下げる。
**手順の概要**: 片方向の参照が不要であることを確認し、逆方向の参照フィールド・更新ロジックを削除する。

### Return Defensive Copy（防御的コピーの返却）
**目的**: 内部の可変状態を外部に直接公開しないようにする。
**手順の概要**: getterで内部コレクション/オブジェクトのコピーを返すよう変更する。

### Immutable Object（イミュータブルオブジェクト化）
**目的**: オブジェクトの状態を生成後に変更不可にし、意図しない副作用を防ぐ。
**手順の概要**: setterを廃止し、状態変更が必要な場合は新しいインスタンスを返すメソッドに置き換える。

## Simplifying Conditional Expressions（条件記述の単純化）

### Decompose Conditional（条件記述の分解）
**目的**: 複雑な条件式と分岐処理をメソッド抽出で読みやすくする。
**手順の概要**: 条件式・then節・else節それぞれを意味の分かるメソッドとして抽出する。

### Replace Conditional with Polymorphism（条件分岐のポリモーフィズムへの置換）
**目的**: 型コードによるswitch/if-else連鎖を、サブクラスのオーバーライドで置き換える。
**手順の概要**: 型ごとのサブクラスを作り、分岐していた各処理をそれぞれのサブクラスのメソッドとして実装する。
```
// Before
function speed(bird) {
  switch (bird.type) {
    case 'European': return 35;
    case 'African': return 40 - bird.numberOfCoconuts;
    case 'NorwegianBlue': return bird.isNailed ? 0 : 10 + bird.voltage;
  }
}

// After
class EuropeanSwallow { getSpeed() { return 35; } }
class AfricanSwallow { getSpeed() { return 40 - this.numberOfCoconuts; } }
class NorwegianBlueParrot { getSpeed() { return this.isNailed ? 0 : 10 + this.voltage; } }
```

### Replace Type Code with State/Strategy（型コードのState/Strategyパターン化）
**目的**: 状態やアルゴリズムを表す型コードを、差し替え可能なオブジェクトに置き換える。
**手順の概要**: 状態/戦略ごとのクラスを作り、コンテキストオブジェクトがそれを保持・委譲するようにする。

### Introduce Null Object（Nullオブジェクトの導入）
**目的**: null チェックの散在を、nullを表す専用オブジェクトへの置き換えで解消する。
**手順の概要**: nullの代わりに何もしない/デフォルト値を返すオブジェクトを用意し、null判定分岐を除去する。

## Simplifying Method Calls（メソッド呼び出しの単純化）

### Introduce Parameter Object（パラメータオブジェクトの導入）
**目的**: 常にセットで渡される引数群を1つのオブジェクトにまとめる。
**手順の概要**: 引数群を保持するクラスを作り、メソッドのシグネチャをそのオブジェクト1つに置き換える。

### Preserve Whole Object（オブジェクト全体の受け渡し）
**目的**: あるオブジェクトから複数の値を取り出して渡す代わりに、オブジェクト自体を渡す。
**手順の概要**: 呼び出し元でオブジェクトから値を取り出す処理をやめ、オブジェクトそのものを引数にする。

### Remove Parameter（不要な引数の削除）
**目的**: メソッド内で使われなくなった引数を除去し、シグネチャを単純化する。
**手順の概要**: 引数の使用箇所がないことを確認し、シグネチャと全呼び出し箇所から削除する。

### Rename Method（メソッド名の変更）
**目的**: メソッドの振る舞いを正しく表す名前に変更し、可読性を上げる。
**手順の概要**: 新しい名前をつけ、全呼び出し箇所を更新する（言語機能のリネームや非推奨化を活用）。

## Dealing with Generalization（一般化の取り扱い）

### Pull Up Method / Pull Up Field（メソッド/フィールドの引き上げ）
**目的**: 複数のサブクラスにある重複を、親クラスに1つだけ定義してまとめる。
**手順の概要**: 各サブクラスの実装が同一であることを確認し、親クラスに移してサブクラス側を削除する。

### Form Template Method（テンプレートメソッドの形成）
**目的**: 手順は同じだが詳細が異なるサブクラスのメソッドを、共通アルゴリズム＋差分フックに整理する。
**手順の概要**: 共通の処理手順を親クラスのメソッドとして定義し、異なる部分を抽象/オーバーライド可能なメソッドとして残す。

### Extract Subclass（サブクラスの抽出）
**目的**: 特定のインスタンスだけが持つ機能を、専用のサブクラスに切り出す。
**手順の概要**: 該当機能のためのサブクラスを作り、関連フィールド・メソッドを移す。

### Extract Interface（インターフェースの抽出）
**目的**: 複数クラスに共通する一部の責任を、インターフェースとして明示する。
**手順の概要**: 共通メソッド群をインターフェースとして宣言し、各クラスに実装させる。

### Collapse Hierarchy（階層の統合）
**目的**: ほとんど差分のないスーパークラス/サブクラスを1つに統合する。
**手順の概要**: サブクラスとスーパークラスのメンバーを1つのクラスにまとめ、片方を削除する。

### Replace Inheritance with Delegation（継承から委譲への置換）
**目的**: 「is-a」ではなく「has-a」の関係であるにも関わらず継承しているものを、委譲に置き換える。
**手順の概要**: 親クラスへの参照をフィールドとして持たせ、必要なメソッドだけを委譲経由で公開する。

## Dealing with Comments（コメントの取り扱い）

### Remove Redundant Comment（冗長コメントの削除）
**目的**: コードを読めば分かる内容の繰り返しを取り除き、保守対象を減らす。
**手順の概要**: コメントが説明している内容とコードを見比べ、コード単体で自明であればコメントを削除する。

### Remove Commented-Out Code（コメントアウトコードの削除）
**目的**: 使われていない旧コードによる混乱を無くし、バージョン管理履歴に判断を委ねる。
**手順の概要**: 無効化されたコードブロックが不要であることを確認し、削除する（復元が必要な場合はコミット履歴から参照する）。

### Extract Method（説明コメントのメソッド抽出）
**目的**: 処理内容をコメントで説明する代わりに、意図を表す名前のメソッドとしてコードそのものに語らせる。
**手順の概要**: コメントが指す処理範囲をメソッドとして抽出し、コメントの内容をそのままメソッド名にする（詳細は[Composing MethodsのExtract Method](#extract-methodメソッドの抽出)を参照）。

### Introduce Assertion（前提コメントのアサーション化）
**目的**: コメントでのみ表現されていた前提条件・不変条件を、実行時に検証可能な形にする。
**手順の概要**: コメントが述べている前提を`assert`文などで明示し、コメントは削除するか要約に留める。

### Replace What-Comment with Why-Comment（WhatコメントのWhyコメント化）
**目的**: コードが語れる「何をしているか」の説明を減らし、コードだけでは分からない「なぜそうしたか」を残す。
**手順の概要**: コメントの内容を確認し、実装内容の説明であれば削除、選択理由・トレードオフ・非自明な制約の説明であれば残す（または書き直す）。
```
// Before
// 合計金額を計算する
let total = items.reduce((sum, item) => sum + item.price, 0);

// After
// 端数処理の都合上、税抜き価格の合計から一括で税額を計算する（レジ側の丸め誤差対策）
let total = items.reduce((sum, item) => sum + item.price, 0);
```
